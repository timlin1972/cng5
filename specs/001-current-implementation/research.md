# Phase 0 Research: 現有系統能力總覽

本文件不是「調查未知選項」，而是把既有程式碼（多半已在原始碼註解中寫明理由）中
已經拍板的關鍵技術決策整理成標準的 Decision/Rationale/Alternatives 格式，作為未來
新功能規劃時比對「這個決定當初為什麼這樣做」的參考基準。spec.md 沒有留下任何
`[NEEDS CLARIFICATION]` 標記，因此本文件的每一節都是「回溯記錄」而非「待決研究」。

## 1. 跨網域訊息加密演算法

- **Decision**: 使用 ChaCha20-Poly1305（AEAD，含完整性驗證），對整個訊息 payload
  加密，而非只加密部分欄位或只做簽章。
- **Rationale**: 單一函式呼叫同時取得機密性與完整性驗證，不需要額外拼裝
  MAC；`chacha20poly1305` crate 在無硬體 AES 加速的環境（如樹莓派等 ARM 裝置）
  效能優於 AES-GCM，適合可能佈署在各種機器上的個人工具。
- **Alternatives considered**: AES-256-GCM（需要硬體加速才有具競爭力的效能，且在
  純軟體實作上比 ChaCha20-Poly1305 慢）；只簽章不加密（無法滿足「訊息內容本身也
  需要保密」的需求，例如檔案內容、指令內容）。

## 2. 防重放機制：時間戳窗口 + 已用 nonce 記錄表

- **Decision**: 每則訊息內嵌 Unix 秒數時間戳（`Envelope.ts`），超出 `±30` 秒即拒絕；
  另外用一張「已成功解密過的 nonce」表（保留 3 倍時間窗）擋掉時間窗內原封不動的
  重放。
- **Rationale**: 只檢查時間戳無法區分「兩個剛好落在同一秒的不同合法訊息」與「同一
  則訊息被重放」——兩者時間戳相同，但合法情況下 nonce 一定不同（每次加密都重新
  產生隨機 nonce）。兩種機制互補：時間戳窗口防止「舊訊息無限期有效」，nonce 表
  防止「窗口內的精確重放」。
- **Alternatives considered**: 只用遞增序號防重放（需要跨機器/跨連線同步一個計數
  器，在「多個裝置可能同時對同一目標發起請求」的場景下難以維護且容易誤判）；
  只檢查時間戳不記錄 nonce（會漏掉「攻擊者在窗口內原封不動重送」的情況）。

## 3. 共用金鑰佈署方式

- **Decision**: 對稱金鑰以純文字 hex 字串存在本機檔案 `remote-key`（64 hex 字元 =
  32 bytes），所有參與跨網域功能的機器手動放置同一份內容；不提供自動產生或分發
  指令，檔案已加入 `.gitignore` 不進版控。
- **Rationale**: 使用情境是使用者本人管理少量自己的機器，手動複製一次性設定檔的
  操作成本遠低於設計一套金鑰分發/輪替機制的複雜度與其引入的新攻擊面（例如金鑰
  分發通道本身的安全性）。與專案既有的其他「手動同步機器設定」慣例一致。
- **Alternatives considered**: 自動化金鑰交換協定（如 Diffie-Hellman 或透過現有
  連線協商）——對單一維運者、少量機器的場景是過度工程，且需要額外的信任建立
  步驟（如何確認對方就是預期的機器）。

## 4. 跨網域訊息大小限制與分段策略

- **Decision**: 檔案傳輸切成 `FILE_CHUNK_SIZE = 4 KiB` 原始位元組一個 chunk（一次
  MQTT 請求/回覆往返一個 chunk）；檔案清單依 `FILE_LIST_PAGE_BUDGET = 4 KiB`
  （以檔名長度估算）分頁, 用回覆帶的 `total` 判斷是否還有下一頁。
- **Rationale**: 實測公開 broker（`broker.emqx.io`）發現單則訊息原始 payload 超過
  約 11000 bytes 會被伺服器端靜默丟棄（不回錯誤、連線不中斷，訊息就是不會送達）。
  這不是憑空假設的限制——曾經因為原本設定的 16 KiB chunk（base64 編碼＋AEAD＋JSON
  包裝後遠超過門檻）導致跨網域檔案複製「永遠卡在逾時，不管把逾時時間調多長都
  一樣」，因為回覆本來就沒有抵達過。4 KiB 原始資料換算後仍遠低於實測門檻，留有
  接近一倍的安全餘裕。
- **Alternatives considered**: 提高逾時時間（無效——問題不是「等得不夠久」，是
  訊息從未送達）；改用可自架、無訊息大小限制的 broker（需要使用者額外維運一個
  對外服務，違背「不需要任何一方對外開放連接埠」的核心價值主張）；把整個檔案
  塞進單一則訊息（同樣會撞上大小門檻，且大檔案完全無法傳輸）。

## 5. 群組拓樸與跨網域橋接機制

- **Decision**: 群組（domain）內用「一台伺服器 + N 台客戶端」的 HTTP 拓樸（客戶端
  每 10 秒回報並拉取完整清單）；多個 domain 之間則透過共同的公開 MQTT broker
  （`broker.emqx.io`）以 `<bridge-id>/<domain>/...` 為 topic 前綴橋接，只有 domain
  的伺服器角色會真的連上 MQTT。
- **Rationale**: 同一區網內用 HTTP 輪詢已經足夠簡單可靠；跨網域（彼此網路不可達）
  的場景才需要一個雙方都連得到的第三方會合點，MQTT publish/subscribe 模式天生
  適合「不知道對方確切位址、只知道一組共同代號」的會合需求。只讓伺服器角色連
  MQTT，是為了讓客戶端不需要額外設定，只需要問自己的伺服器要資料即可，降低每台
  客戶端機器的設定負擔。
- **Alternatives considered**: 每台機器都直接連 MQTT（增加客戶端設定複雜度，且
  沒有實際好處——反正客戶端本來就已經有一個可以問的伺服器）；使用 P2P 打洞技術
  （NAT traversal 在个人工具的場景下複雜度與 MQTT 中繼相比不成比例，且不是所有
  網路環境都能成功打洞）。

## 6. 自我更新（upgrade）流程設計

- **Decision**: `upgrade` 指令依序執行 `git fetch --all`、`git reset --hard
  origin/main`、`cargo build`；編譯期間舊行程持續運作，只有編譯成功才會重新呼叫
  自己（`--respawn-after=<pid>`）等舊行程真正結束、放開 port 後才啟動新執行檔；
  提供 `CNG5_EXTERNAL_RESTART` 環境變數讓外部已有的行程監督機制（如常駐守護
  script）接手重啟，避免兩邊搶著啟動、搶 bind port。
- **Rationale**: 個人維運情境下，使用者常常不在機器前，服務中斷視窗必須盡量小；
  「先確認編譯成功再切換」避免把機器更新到一個編譯失敗、什麼都跑不起來的狀態。
  透過既有的 `execute_line` 指令路徑觸發，代表這個指令跟其他指令一樣可以透過
  `remote`/跨網域連線在遠端機器上執行，不需要另外設計一套「遠端更新」協定。
- **Alternatives considered**: 下載預編譯好的執行檔（需要額外維護建置與發布管線，
  對單一維護者、原始碼即真相的專案不成比例）；容器化部署（超出這個工具「單一
  執行檔＋設定檔」的定位，也不適合它要直接存取的作業系統層資訊，如 ARP 表、
  Tailscale 狀態）。

## 7. 三種前端共用同一份狀態

- **Decision**: CLI、GUI、瀏覽器三種介面都呼叫同一個 `Shell::execute_line` 與同一
  份 `ContextInner`/`OutputBuffer`，不各自維護一份獨立狀態。
- **Rationale**: 使用者可能在任何時刻切換操作介面（例如手機掃 QR Code 開瀏覽器、
  同時終端機還開著），如果三者狀態不同步，會讓使用者對「機器目前到底是什麼狀態」
  失去信任，這對一個維運工具是不可接受的。
- **Alternatives considered**: 每種介面各自維護狀態、定期同步（引入額外的同步延遲
  與不一致視窗，且沒有必要——三者本來就在同一個行程裡，可以直接共用同一份記憶體
  中的狀態）。

## 8. 版本號管理

- **Decision**: `APP_VERSION` 是寫死在原始碼裡的字串常數（目前 `"1.3.0"`），不是
  從 `Cargo.toml`／git tag 動態讀取。
- **Rationale**: 裝置回報（`DeviceReport.version`）、`system version` 指令、
  `/api/version` 都讀這一個常數；發新版本時只需要改這一行，比起在建置流程中注入
  git tag 或解析 `Cargo.toml`（需要額外的建置腳本/build.rs 邏輯）更直接，符合單一
  維護者的精簡原則。
- **Alternatives considered**: 建置時從 git tag 自動產生版本號（對只有一個人維護、
  發版頻率由自己掌控的專案，額外的建置複雜度換不回實質好處）。
